# Google OAuth PKCE Re-Auth Failure

**Experience:** 29.05.2026 â€” After token expired/revoked, the `setup.py --auth-url` + `--auth-code` flow 
kept failing with:

```
ERROR: Token exchange failed: (invalid_grant) code_verifier or verifier is not needed.
```

And the manual curl equivalent:

```json
{"error": "invalid_grant", "error_description": "Bad Request"}
```

## Root Cause

The PKCE challenge **IS** correctly computed (sha256 of the stored verifier matches the URL's 
`code_challenge` parameter). Google rejects the exchange anyway. Likely causes:

1. **Confidential client conflict** â€” The OAuth client is a "Desktop app" type with a 
   `client_secret`. PKCE was designed for **public clients** (no secret). Sending BOTH 
   `client_secret` AND `code_verifier` can cause Google to reject with `invalid_grant`.
2. **Code expiry** â€” Authorization codes are only valid for ~5 minutes. Any delay between 
   the user visiting the URL and the agent calling `--auth-code` can expire it.
3. **Verifier format issues** â€” The `code_verifier` generated by `setup.py` contains `~` and 
   `.` characters. While RFC 7636 allows unreserved characters, some Google endpoints are 
   picky about the exact format.

- Manual curl with `code_verifier` â†’ `{"error": "invalid_grant"}`

## Mitigation Options When `--auth-code` Fails

### Option A: Try exchange without code_verifier

For confidential clients, PKCE is optional. Try the exchange without a code_verifier:

```bash
CLIENT_ID=$(python3 -c "import json; print(json.load(open('/root/.hermes/google_client_secret.json'))['installed']['client_id'])")
CLIENT_SECRET=$(python3 -c "import json; print(json.load(open('/root/.hermes/google_client_secret.json'))['installed']['client_secret'])")

CODE="CODE_FROM_USER"
curl -s -X POST https://oauth2.googleapis.com/token \
  -d "code=$CODE" \
  -d "client_id=$CLIENT_ID" \
  -d "client_secret=$CLIENT_SECRET" \
  -d "redirect_uri=http://localhost:1" \
  -d "grant_type=authorization_code"
```

### Option B: Use the refresh token directly on the existing token

If `google_token.json` still has a `refresh_token`, try forcing a refresh:

```python
from google.oauth2.credentials import Credentials
from google.auth.transport.requests import Request
creds = Credentials.from_authorized_user_file('/root/.hermes/google_token.json')
creds.refresh(Request())
```

May still fail with `invalid_grant`, but sometimes a second attempt hours later succeeds.

### Option C: Full reset = new client_secret

If all above fails, create a brand new OAuth 2.0 Client ID (Desktop app type) in 
Google Cloud Console and use the new `client_secret.json`. This bypasses any corrupted 
state in the old client.

### Option D: Fallback to Gmail App Password via Himalaya

If the goal is just email, skip OAuth entirely. Use Himalaya with a Gmail App Password:
1. Google Account â†’ Security â†’ App Passwords
2. Generate for "Mail" on "Other" device
3. Configure `~/.config/himalaya/config.toml` with IMAP + SMTP + that password
4. No OAuth needed, no expiry issues
